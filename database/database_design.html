核心表详解

3.1 documents（文件主表）— 系统核心

所有业务围绕此表展开，是文件管理、OCR提取、审批流、审计日志的中心节点。

关联关系
document_type_id → document_types.id
order_id → orders.id
customer_id → customers.id
uploaded_by → users.id
reviewed_by → users.id
approved_by → users.id
状态流转
draft（草稿）
→ ocr_processing（OCR处理中）
→ pending_review（待核对）
→ pending_approval（待审核）
→ approved（审核通过）→ archived（已归档）
→ voided（已作废，任意状态可触发）
设计要点：采用软删除（deleted_at），文件不会物理删除。file_hash 用于去重检测。FULLTEXT 索引支持文件编号和标题的模糊搜索。

3.2 document_fields（字段提取值表）— 数据资产核心

存储 OCR/NLP 从每个文件中提取的结构化字段值，是未来大模型训练数据的源头。

字段元信息
field_key — 冗余字段标识，避免每次 JOIN 模板表
confidence — OCR 置信度，低于阈值自动进入核对队列
bbox_info — JSON 存储 OCR 定位坐标，用于前端高亮标注
extract_method — 区分自动提取/人工录入/推断填补
数据流
1. Python OCR 提取后写入此表
2. 低置信度字段生成 review_task
3. 人工核对后更新 is_confirmed
4. 字段修改记录写入 field_change_logs
5. 清洗后的数据供大模型训练使用

3.3 field_templates（字段模板表）— 可配置化基础

每种文件类型对应一组字段模板，管理员可在系统设置中自定义字段，无需改代码。

关键字段作用示例值
field_key字段唯一标识，代码中引用declaration_no, trade_amount
field_type字段数据类型，控制校验和展示string / decimal / date / enum
is_required是否必填，空值时标记为待核对1（必填）/ 0（可选）
extract_ruleOCR 提取规则（正则或位置模板）^\d{18}$（报关单号格式）
validation_rule入库校验规则digits:8,10（HS编码位数）
enum_options枚举字段的可选值（JSON）["一般贸易","加工贸易","保税"]

3.4 orders（订单表）— 业务关联锚点

订单是串联所有文件的主线。一个订单（如 SO-8820）可关联报关单、发票、箱单、提单等多个文件，实现"一票货全套单据"的管理。

3.5 approval_records + approval_details — 多级审批

两表协作实现可配置的多级审批：

approval_records
记录一次完整的审批流程（谁提交、当前第几级、最终结果）
approval_details
记录每一级审批人的具体动作（通过/驳回、审批意见、操作时间）
4
核心数据流

4.1 文件上传 → OCR → 归档 全流程

1. 用户上传文件 → 写入 documents (status=draft) → 文件存储到磁盘 (storage_path) → 写入 document_versions (version=1) 2. 提交 OCR → documents.status → ocr_processing → 创建 ocr_tasks (status=pending) → Redis 队列通知 Python Worker 3. Python OCR 处理 → 更新 ocr_tasks.status → processing → completed → 按 field_templates 提取字段 → 写入 document_fields (含 confidence) → 写入 pipeline_logs (stage=ocr) 4. 判断是否需要人工核对 → 若存在 confidence < 阈值的字段: → documents.status → pending_review → 创建 review_tasks → 创建 notifications (通知单证员) → 若全部字段置信度达标: → 直接进入步骤 5 5. 人工核对完成 → 更新 document_fields.is_confirmed = 1 → 写入 field_change_logs (如有修改) → documents.status → pending_approval 6. 审批流 → 创建 approval_records → 创建 notifications (通知审批人) → 审批人操作 → 写入 approval_details → 通过: documents.status → archived → 驳回: documents.status → pending_review 7. 跨文件校验 (异步) → 比对同 order_id 下的文件字段 → 发现异常 → 写入 anomaly_alerts → 创建 notifications (通知相关人员)

4.2 数据隔离查询逻辑

角色查询 documents 时的过滤条件
超级管理员无过滤，查看全部
业务主管WHERE uploaded_by IN (SELECT id FROM users WHERE department_id = ?)
业务员WHERE customer_id IN (SELECT customer_id FROM user_customers WHERE user_id = ?)
单证员同业务主管（部门级）
财务WHERE document_type_id IN (银行水单, 商业发票)
只读访客按临时授权的文件 ID 列表过滤

5.1 关键索引说明

表索引用途
documentsFULLTEXT (doc_no, title, original_filename)文件搜索框的模糊搜索
documentsidx_status + idx_trade_date + idx_created_at文件列表筛选排序
documentsidx_hash (file_hash)上传去重检测
document_fieldsidx_document + idx_confidence按文件查字段 + 低置信度筛选
ocr_tasksidx_status_priorityWorker 拉取待处理任务（按优先级）
operation_logsidx_created_at + idx_target审计日志按时间范围查询
notificationsidx_user_read用户未读通知列表
customersFULLTEXT (company_name, company_name_en, short_name)客户名搜索

5.2 大表预估与分区建议

表预估年增长建议
documents~5,000-20,000 行/年初期无需分区，百万级以内 MySQL 单表无压力
document_fields~50,000-200,000 行/年使用 bigint 主键，预留增长空间
operation_logs~100,000-500,000 行/年超过 500 万行后可按月分区，旧数据归档
pipeline_logs~50,000-200,000 行/年同上，保留 1 年后归档
notifications~100,000+ 行/年已读超 90 天的可定期清理
内部系统性能建议：50 人以内使用，年文件量万级，MySQL 单库单表完全胜任，不需要分库分表。重点关注：1) 文件存储磁盘空间定期监控；2) operation_logs 表定期归档；3) Redis 内存不超限。
